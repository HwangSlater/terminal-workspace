# Implementation Plan - Phase 4: Release Packaging & Installers

Phase 3 closed with the storage engine reconsidered (`redb`, no C toolchain needed — ADR-0014) and `cargo check`/`clippy`/`fmt` fully clean across all 17 crates. That work made *building from source* easy. It did nothing for the actual goal restated just now:

> **A first-time user should not have to do anything beyond running the app to use it.**

Today, "running the app" still means "have Rust installed, clone the repo, `cargo run -p app`." That's a developer workflow, not a user one. Phase 4 closes that gap: a user downloads one file for their OS and runs it — no Rust, no `cargo`, no toolchain, ever visible to them. This directly completes `docs/01-product/product-requirements.md` §2.1 item 4 ("Prebuilt release binaries") and §4's v1.0.0 scope line, both already committed to earlier in this project's docs — Phase 4 is where that commitment gets built.

This is a **design document for review, not a done deal** — nothing described below has been implemented yet.

---

## User Review Required

> [!IMPORTANT]
> **Key decisions needing sign-off before implementation starts:**
> 1. **Packaging tool: `cargo-dist`**, not a hand-rolled GitHub Actions release job.
> 2. **Installer shape per OS**: Windows `.msi` (or a self-contained `.exe`), macOS `.dmg` (+ optional Homebrew tap later), Linux shell-installer script + tarball (+ optional `.deb`/AppImage later).
> 3. **Release trigger**: pushing a git tag (`v0.1.0` etc.), reusing the existing 3-OS CI matrix rather than building a second parallel pipeline.
> 4. **Explicitly out of scope for Phase 4**: auto-update, code signing/notarization (flagged as a real gap — unsigned installers trigger OS security warnings; see Risks), package manager submissions (winget/homebrew-core/apt repos) beyond a self-hosted tap/manifest.

---

## Why `cargo-dist`

| | `cargo-dist` | Hand-rolled `release.yml` |
| :--- | :--- | :--- |
| **Effort** | Config file + `cargo dist init`; generates the GitHub Actions workflow itself | Write and maintain every OS's build/package/upload steps by hand |
| **Output** | `.msi`, tarballs, a `curl \| sh` shell installer *and* a PowerShell installer, a GitHub Release with checksums — out of the box | Whatever we build, one artifact type at a time |
| **Maintenance** | Upstream-maintained; new Rust/OS quirks get fixed centrally | We carry every fix ourselves |
| **Ecosystem fit** | Widely used across the Rust CLI ecosystem already (so contributors likely already know the shape of it) | Bespoke, one more thing to document from scratch |

Hand-rolling remains an option if `cargo-dist`'s conventions don't fit something we need later — but there's no concrete reason yet to take on that maintenance cost instead of the tool built for exactly this job. Recommendation: `cargo-dist`.

## Proposed Changes

### Config
#### [NEW] `Cargo.toml` (workspace) — `[workspace.metadata.dist]` section
- Declares `crates/app`'s binary (`terminal-workspace`) as the release artifact, target platforms (`x86_64-pc-windows-msvc`, `x86_64-apple-darwin`, `aarch64-apple-darwin`, `x86_64-unknown-linux-gnu`), and installer types (shell, powershell, msi).

#### [NEW] `.github/workflows/release.yml`
- Generated by `cargo dist init` / `cargo dist generate-ci`, not hand-written. Triggered on `v*` tag push. Builds all Tier 1 targets (reusing what `docs/06-development/platform-support.md` already defines), packages installers, publishes a GitHub Release.
- Kept separate from `.github/workflows/ci.yml` (push/PR checks) — different trigger, different purpose, no reason to entangle them.

### Docs (updated before/alongside the workflow, per `docs/06-development/development.md` §2)
- **New ADR**: `docs/06-development/decisions/0015-release-packaging.md` — records the `cargo-dist` choice, the alternatives table above, and the signing/notarization gap as a known limitation.
- `docs/01-product/product-requirements.md` §2.1 item 4 and §4 (v1.0.0 scope): update from "target: v1.0.0" aspirational language to point at the concrete mechanism once it exists.
- `docs/01-product/roadmap.md`: same.
- Root `README.md`: add a **Download** section above Quick Start — "grab the installer for your OS from the latest release" as the *first*, primary path; `cargo run -p app` demoted to a "building from source" subsection for contributors.
- `docs/06-development/platform-support.md`: cross-reference the release matrix so it's clear Tier 1 support and release-artifact support are the same three targets, not two lists that can drift apart.

## Sequencing (decided)

`crates/app` is currently a boot-and-exit skeleton (Phase 2/3 vertical-slice proof, not an interactive TUI — `docs/02-architecture/ui.md`'s docking system isn't implemented yet). Decision: **build the packaging pipeline against the skeleton now, TUI after.**

This is a deliberate "walking skeleton" / steel-thread sequencing choice: prove the full build → package → install → run pipeline end-to-end while the payload is trivial, so that signing/OS-packaging/CI quirks get found and fixed now rather than entangled with TUI debugging later. Concretely:

1. **Phase 4 (this doc)**: `cargo-dist` pipeline, validated only via pre-release tags (e.g. `v0.0.1-test`). **No public GitHub Release, no README Download section yet** — this phase proves the mechanism, it does not announce a product.
2. **Phase 5 (next, not this doc)**: TUI implementation (`docs/02-architecture/ui.md` docking system, `docs/02-architecture/keyboard.md` bindings).
3. **Only after Phase 5**: a real public release and the README `Download` section go live, pointing at a build that's actually worth downloading.

## Risks / Open Questions (still open — flagging, not deciding unilaterally)

- **Unsigned binaries**: Windows SmartScreen and macOS Gatekeeper both warn on unsigned installers from an unknown publisher. This won't fully disappear without a paid code-signing certificate (and, for macOS, Apple Developer Program notarization). Phase 4 as scoped ships *unsigned* installers — first-run shows an OS warning the user has to click through once. Since Phase 4's installers are pre-release/internal-validation only (per the sequencing above), this is lower-stakes than it would be for a real public release — but the Phase 5+ public release will need an actual answer, not just a deferral. Worth deciding now: accept the warning long-term (many small open-source CLI tools do), or budget for a signing certificate before the real v1.0.0 public release.

  **Resolved, right before the actual v1.0.0 tag** (pre-release readiness sweep, decided directly with the user): accept the warning long-term, no signing certificate for this release. Recorded in `CHANGELOG.md`'s "Known limitations" so it's visible to anyone downloading the release, not just to someone reading this implementation log. Revisit if it turns out to be a real adoption blocker.

## Verification Plan

- Push a pre-release tag (e.g. `v0.0.1-test`) to confirm the release workflow produces working installers for all three Tier 1 OSes.
- On a machine with **no Rust toolchain installed at all**, download and run the generated installer, then run the resulting `terminal-workspace` command — this is the actual test of the stated goal, not `cargo check` passing.
- Confirm checksums/artifact list match `cargo-dist`'s manifest so a future auto-update mechanism (out of scope now) has something correct to build on.

## Implementation Notes (what actually happened)

- Repository URL decided: `https://github.com/HwangSlater/terminal-workspace` (not yet pushed — `git init` was local-only; no remote configured, no push performed).
- `cargo-dist 0.32.0` installed via its prebuilt-binary shell installer (no local compilation — sidesteps this machine's separate, unrelated linker issue entirely).
- `dist init -y --ci github --installer shell --installer powershell --installer msi --target x86_64-pc-windows-msvc --target x86_64-apple-darwin --target aarch64-apple-darwin --target x86_64-unknown-linux-gnu` generated `dist-workspace.toml`, `[profile.dist]` in the root `Cargo.toml`, `.github/workflows/release.yml`, and `crates/app/wix/main.wxs`.
- `dist init`'s recommended defaults included `aarch64-unknown-linux-gnu` — trimmed back out of `dist-workspace.toml` to match `docs/06-development/platform-support.md`'s Tier 1 list exactly, rather than silently expanding platform scope through a tool default.
- `crates/app/Cargo.toml` gained `description` and `repository.workspace = true` (required by `cargo-dist`; also just good `Cargo.toml` hygiene that was missing before).
- `dist plan` confirms a coherent release plan (tarballs, `.msi`, shell/PowerShell installers, checksums for all four targets) without needing an actual tag push to validate the config shape.
- Full record and rationale: `docs/06-development/decisions/0015-release-packaging.md`.
- **Not done**: pushing to the actual GitHub remote, cutting a real (or pre-release) tag, or verifying the CI workflow runs end-to-end on GitHub's runners — all of that requires the user's own GitHub push, which wasn't part of this pass.
