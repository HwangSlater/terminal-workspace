# Implementation Plan - Phase 14: Plugin Runtime (v0.5)

This is a **design document for review — nothing described below has been implemented yet**, per the same process used for Phases 6-13.

## Context

Unlike every integration phase so far, the plugin architecture was substantially designed *before* this project's implementation-first phases even began: ADR-0002 (WASM via `wasmtime`, capability sandboxing, fuel/memory limits), ADR-0009 (WebAssembly Component Model + WIT over raw-pointer FFI, for type safety), `docs/04-extensions/plugins.md` (host-guest FFI shape), `docs/04-extensions/plugin-lifecycle.md` (state machine, fuel budget of 1,000,000 instructions per event, 64MB memory limit), and `docs/04-extensions/plugin-sdk.md` (a concrete draft `plugin-sdk.wit` — `host-services` interface with `publish-event`/`log`/`get-member-presence`, a `developer-workspace-plugin` world exporting `initialize`/`on-event`/`shutdown`). None of this exists as code yet: `crates/plugin-host` and `crates/plugin-sdk` are 47 and 30 lines of stub respectively, no `.wit` file exists on disk, and neither crate has a WASM runtime dependency.

**A real finding that required stopping and confirming with the user before writing anything further**: `wasmtime` — ADR-0002's named runtime — pulls in the `cc` crate as a direct build-dependency (confirmed via `cargo tree -p plugin-host -i cc` after a real `cargo add`, not assumed). This contradicts ADR-0014's "no crate in this workspace needs a C compiler" principle, which every phase since has been built to preserve (the Phase 6 `rustls`/`ring` swap to `native-tls`, the Phase 12 `ical`/`rrule` verification that both are pure Rust before committing to them). The pure-Rust alternative, `wasmi`, verified clean (no `cc` anywhere in its tree) but has no Component Model/WIT support, which would mean re-opening ADR-0009 too and falling back to the raw-pointer/JSON FFI boundary ADR-0009 explicitly rejected as unsafe. **Confirmed with the user: stay on `wasmtime`, accept the C-compiler requirement for contributors building from source.** This is a real, scoped exception to ADR-0014 — not a reversal of it — and needs its own ADR (Decision 1 below) rather than silently contradicting a standing principle undocumented. Critically, this only affects **contributors building from source**; end users of prebuilt release binaries (`cargo-dist`, ADR-0015) are entirely unaffected — `wasmtime`'s JIT (Cranelift) is statically linked into the shipped binary and compiles guest `.wasm` at runtime without ever invoking an external compiler.

**Given the size of this phase relative to every prior one** (a real sandboxed runtime, a new interface-definition toolchain, a new guest-side build pipeline — not "poll an HTTP API on a timer"), this doc scopes deliberately narrow: prove the sandbox/lifecycle/resource-limit plumbing end-to-end with one minimal example plugin, and defer plugin-registered commands/UI panels (ADR-0010's unified registries, already implemented and sitting unused — `InMemoryUiRegistry` is already constructed in `crates/app/src/main.rs` and handed to `TuiRenderer` with `#[allow(dead_code)]`, "wired for future dynamic panel registration") to a follow-up phase. The draft `plugin-sdk.wit`'s `world` only exports `initialize`/`on-event`/`shutdown` anyway — no command/panel registration exports exist in it yet, so extending the WIT surface for that is itself new design work, not a checkbox in this one.

## Decisions

### 1. WASM runtime: `wasmtime`, with a new ADR documenting the C-compiler exception

**Confirmed**: stay on `wasmtime` (ADR-0002/0009 unchanged), write **ADR-0017** documenting the C-compiler-for-contributors exception explicitly — same spirit as ADR-0014 itself (a storage-engine reconsideration) and the Phase 6/7 Linux-linker amendment to `platform-support.md`, but framed as an *accepted, scoped* exception rather than a reversal: the plugin runtime is the one place in this workspace where a C compiler is genuinely necessary (WASM sandboxing has no pure-Rust option with Component Model maturity), the cost lands only on contributors building from source, and it's avoidable entirely by not enabling the (default-off, per Decision 4) plugin system during development if a contributor isn't working on that area. `docs/06-development/platform-support.md` gets a "Phase 14" amendment alongside it, matching the existing update-note pattern.

### 2. Guest plugin build tooling

**Confirmed**: `cargo-component` — the Bytecode Alliance's standard cargo subcommand that wraps "compile to `wasm32-wasip1`/`wasm32-unknown-unknown` + generate WIT bindings + componentize" into `cargo component build`. Plugin authors install it once (`cargo install cargo-component`); it's a plugin-author-side tool, not something every `terminal-workspace` contributor needs (only contributors actually authoring/testing a plugin do). It's the ecosystem-standard path and keeps the guest side of this phase's actual new work small — the interesting, risky part is the host side (real sandboxing, real resource limits), not re-deriving guest tooling ADR-0009 already pointed at (`wit-bindgen`).

### 3. Host function surface for this phase

**Confirmed**: exactly the two lowest-risk functions from the existing `plugin-sdk.wit` draft — `log(level, message)` (proves the plugin is alive, zero side effects beyond `tracing`) and `publish-event(event-type, payload)`, mapped to the already-frozen `Event::PluginCustomEvent { plugin_id, payload_json }` (the host injects `plugin_id` from which `Store` is calling; `event-type`+`payload` get JSON-enveloped into `payload_json` as `{"event_type":.., "payload":..}` since the frozen variant has no separate `event_type` field — no `Event` enum change needed, no ADR). `get-member-presence` (a real read against `PresenceRepository`) and every `PluginCapability` variant (`NetworkConnect`/`FsRead`/`FsWrite`/`SlackRead`) are left **defined but unenforced** — nothing in this phase's example plugin needs them, and wiring real enforcement for capabilities nothing yet exercises would repeat the exact "don't build for hypothetical future needs" mistake this project has otherwise avoided. `PermissionManager::verify_capability` stays a stub returning `true` (documented honestly as such, same pattern as `IntegrationAdapter`'s early stubs before Phase 6) until a real plugin actually needs to be gated — matching how `SecretProviderChain`'s keyring backend wasn't built until Phase 7 needed it, not spans of time before.

### 4. Default-off, explicit opt-in

**Confirmed (not separately asked — low-stakes, entailed by Decision 1)**: `[plugins]` stays `enabled`-gated and effectively unused unless a contributor points `directory` at a real plugin and adds it to `allowed_list`, mirroring every integration's `enabled` flag. Nothing about plugin support runs, loads, or requires the C-compiler-built `wasmtime` crate to do anything at runtime just because the workspace compiles it in — this only matters for anyone actually exercising the feature.

## Proposed Changes (pending confirmation of Decisions 2-3 above)

#### [NEW] `docs/06-development/decisions/0017-plugin-runtime-c-compiler-exception.md`
Documents Decision 1 in full ADR form: context (the `cc`-as-build-dependency finding), decision (accept it, scoped to the plugin runtime), alternatives considered (`wasmi`, deferring plugins further), consequences (Contributor Experience-only impact, `platform-support.md` amendment, per-OS setup notes below).

#### [MODIFY] `docs/06-development/platform-support.md`, `README.md`
A "Phase 14" update note (matching the Phase 6/7 Linux-OpenSSL note's shape) stating a C compiler is now genuinely required to build `crates/plugin-host` specifically — Windows: MSVC Build Tools or the already-documented WinLibs MinGW; macOS: Xcode Command Line Tools (already required, no new step); Linux: `build-essential`/equivalent (previously only a linker was needed — this is Linux's first *real* C-compilation requirement in this project). `scripts/setup.sh`/`setup.ps1` detection updated to check for an actual C compiler, not just a linker, on Linux.

#### [NEW] `crates/plugin-sdk/wit/plugin-sdk.wit`
The actual `.wit` file — content matching `docs/04-extensions/plugin-sdk.md`'s draft, trimmed to this phase's host-function scope (Decision 3): `host-services` interface with `log`/`publish-event` only (drop `get-member-presence` and the `notification`/`presence-status` types nothing yet uses); `developer-workspace-plugin` world unchanged (`initialize`/`on-event`/`shutdown` exports).

#### [MODIFY] `crates/plugin-sdk/src/lib.rs`
Replace the current plain `#[async_trait] GuestPlugin` trait (a native-Rust shape that was never going to be what a sandboxed WASM guest actually implements) with `wit_bindgen::generate!` bindings generated from the new `.wit` file — this becomes the real guest-side SDK a plugin author's `cargo-component`-built crate depends on. `PluginCapability` enum stays as manifest-shape (Decision 3), used by the plugin's manifest/Cargo metadata, not enforced yet.

#### [MODIFY] `crates/plugin-host/src/lib.rs`
- `wasmtime::component::bindgen!` generates the host-side bindings from the same `.wit` file.
- `PluginHostManager` gains real behavior: discover `*.wasm` components in `[plugins].directory` filtered by `allowed_list`, `wasmtime::Engine`/`Store` per plugin instance with fuel (1,000,000/event, `plugin-lifecycle.md`) and memory (64MB) limits configured exactly as already documented, instantiate, call `initialize`, forward `EventBus` events into `on-event` (JSON-serialized, matching `docs/02-architecture/events.md`'s existing "all events are serialized to JSON before... dispatch to WASM plugins" line), call `shutdown` on workspace exit.
- Implements the two host-services functions from Decision 3: `log` → `tracing`, `publish-event` → `Event::PluginCustomEvent` via the shared `EventBus`.
- A trapped/panicking guest (fuel exhaustion, out-of-memory, a genuine bug) is caught, logged, and the plugin instance is marked failed — it does not take down the host process. Verified with a real trap-inducing example, not just asserted.

#### [MODIFY] `crates/config/src/lib.rs`
`PluginsSettings { directory: PathBuf, allowed_list: Vec<String> }` (already sketched in `configuration.md` §1, not yet in the real struct), `#[serde(default)]` throughout, same lesson every prior integration's config addition already learned.

#### [MODIFY] `crates/app/src/main.rs`
Construct `PluginHostManager`, call `initialize()`/discovery/lifecycle at the same boot point `docs/02-architecture/architecture.md` already sketches, wire it to the already-constructed `event_bus` and the already-constructed-but-unused `ui_registry`/`InMemoryCommandRegistry` (constructing the latter for the first time — it doesn't exist in `main.rs` today).

#### [NEW] `examples/plugins/hello/` (or similar)
One minimal example plugin proving the pipeline end-to-end: `initialize` logs a greeting via the `log` host function, `on-event` logs which event it received, `shutdown` logs a goodbye. No commands, no UI, no capabilities requested — deliberately the smallest thing that proves load → lifecycle → host-function-calls → resource limits all work for real.

## Verification Plan

- Unit tests for the config-side `PluginsSettings` parsing (mirrors every prior integration's `#[serde(default)]` round-trip tests).
- `PluginHostManager` tests against the real example plugin's compiled `.wasm`: loads and initializes successfully; `on-event` is actually invoked when an `Event` is published on the shared bus; `log`/`publish-event` host calls produce the expected `tracing` output / `Event::PluginCustomEvent`, respectively.
- A **deliberately malicious/broken** second example plugin (infinite loop) proving the fuel limit actually traps execution rather than hanging the host — this is the single most important test in the phase, since "a plugin crash must not crash the workspace" (ADR-0002) is the entire point of choosing WASM over dynamic libraries in the first place.
- Memory-limit test: a plugin that tries to allocate past 64MB traps rather than growing unbounded.
- No live end-to-end TUI test (plugins don't register UI in this phase, per Decision 3's scope) — manual verification: run with a real plugin directory configured, confirm the example plugin's log lines appear in the tracing output.

## Implementation Notes (what actually happened)

Every Proposed Change above was implemented and verified. `cargo fmt --all --check`, `cargo check --workspace --all-targets`, `cargo clippy --workspace --all-targets -- -D warnings`, and `cargo test --workspace` all pass clean on the native `x86_64-pc-windows-gnu` target, including three real `.wasm` components built with `cargo component build` and loaded by the real `PluginHostManager` (not mocked). Several things surfaced only by building it for real, in the order they were hit:

1. **`wit_bindgen::generate!`'s guest-side output type-checks on a native (non-wasm32) target.** This was a genuine open question going in — resolved empirically via `cargo check -p plugin-sdk`, which compiled clean. One guess was wrong along the way: for a WIT world that exports functions *directly* (not through a named interface), the generated `Guest` trait and `export!` macro land at the crate root, not nested under `exports::` — that nesting only applies to exported *interfaces*. Imported interfaces (`host-services`) do nest under their package/interface path.

2. **`wasmtime::component::bindgen!`'s generated code collides with a crate-root `use common::Result`.** The macro's expansion refers to a bare, unqualified `Result` internally; `common::Result<T>` (a 1-generic-argument alias) shadowed it at crate-root scope, producing `E0107: type alias takes 1 generic argument but 2 generic arguments were supplied`. Fixed by scoping the `bindgen!` call into its own `mod bindings` inside `crates/plugin-host/src/lib.rs`.

3. **`wasmtime` 46's `add_to_linker` needs an explicit `HasSelf<_>` type parameter.** `DeveloperWorkspacePlugin::add_to_linker(&mut linker, |state| state)` alone hit `E0283: type annotations needed` — newer wasmtime versions use a `HasData`/`HostWithStore` generic-projection pattern instead of the older two-argument-closure-only API. Fixed by following the error's own suggestion and wasmtime's `bindgen_examples` module (found in the local `cargo` registry cache, not guessed): `add_to_linker::<_, HasSelf<_>>(&mut linker, |state| state)`.

4. **The single most important architectural finding of this phase**: `cargo-component` generates its **own** per-crate bindings (`src/bindings.rs`, from `[package.metadata.component]`'s `target.path`/`target.world`) — it does not use a shared "SDK" crate's own `wit_bindgen::generate!` output. The first version of `examples/plugins/hello` depended on `crates/plugin-sdk` and called `plugin_sdk::export!(HelloPlugin)`, expecting to reuse plugin-sdk's generated `Guest`/`export!`. This failed with `E0603: macro `export` is private` — `wit_bindgen::generate!` only makes `export!` a crate-root item, not `#[macro_export]`ed, and `cargo-component` separately regenerates the same shapes from the WIT file directly into each plugin crate it builds. Consequence: `crates/plugin-sdk` is not a build dependency of `cargo-component`-built plugins. It remains useful as the crate that verified the WIT contract type-checks early (finding 1 above) and as a future home for a real `PluginCapability`-consuming manifest helper, but every real guest plugin in this repo (`examples/plugins/hello`/`looper`/`memhog`) depends only on `wit-bindgen-rt` (the runtime support crate `cargo-component`'s generated bindings need) and points its own `[package.metadata.component]` at the same `crates/plugin-sdk/wit/plugin-sdk.wit` file. `docs/04-extensions/plugin-sdk.md` and `plugins.md` were amended with Implementation Status notes covering this and the earlier raw-pointer-FFI/native-trait drafts they still contained.

5. **`cargo-component`'s default target (`wasm32-wasip1`) pulls in WASI imports the host never provides.** The first real `load_all()` attempt against a `wasm32-wasip1`-built `hello.wasm` failed with `Plugin fault: instantiate hello: component imports instance `wasi:cli/environment@0.2.3`, but a matching implementation was not found in the linker` — the WASI Preview 1 adapter `cargo-component` links in by default expects `wasmtime_wasi::add_to_linker_sync` wiring this project's `Linker<PluginState>` never adds (deliberately: the WIT world imports only `host-services`, no `wasi:*` interfaces). Fixed by building with `cargo component build --target wasm32-unknown-unknown` instead, which produces a component whose only imports are the ones this host actually provides — also the more correct sandboxing default for a plugin that shouldn't have ambient env/clock/fs access per Decision 3's minimal host-function scope.

6. **The fuel and memory limits were verified against real traps, not asserted.** `examples/plugins/looper` (`on-event` busy-loops forever) and `examples/plugins/memhog` (`on-event` allocates 200MB against a 64MB `StoreLimits` ceiling) are real, separately-built `.wasm` components loaded by `PluginHostManager` in `crates/plugin-host/src/lib.rs`'s test suite. Both traps were observed with real wasm backtraces in `tracing` output (`looper.wasm!...on_event`, `memhog.wasm!...alloc_zeroed`), and in both cases the plugin instance was actually removed from `PluginHostManager`'s loaded set afterward — proving the "a plugin crash must not crash the workspace" (ADR-0002) claim end-to-end rather than by code inspection.

7. **`cargo-component` (`cargo install cargo-component --locked`) is a genuinely large build** (~5-10 minutes on this machine) — expected and consistent with `step14.md`'s framing of it as a plugin-author-side, not every-contributor, tool; it was installed once in this environment to build and verify the three example plugins, matching the README's guidance that only contributors actually authoring/testing a plugin need it.

8. **Host-side `publish-event` bridges a synchronous Component-Model call to the async `EventBus::publish`** via `tokio::runtime::Handle::current().block_on(...)`, safe specifically because every guest call in `crates/plugin-host` runs inside a `tokio::task::spawn_blocking` closure (`load_one`) or, for `on-event`/`shutdown`, is itself a bounded (fuel-limited) synchronous call — never directly on a bare Tokio worker thread outside of one. This was a deliberate scope decision (matching Decision 3's "no async host functions this phase") rather than adopting `wasmtime`'s `async_support`/`call_async` machinery, which would have added a second host-function-calling convention for no capability this phase's example plugins need.

Deferred, as scoped by the Decisions above: `get-member-presence` and real `PluginCapability` enforcement (`PermissionManager::verify_capability` remains a stub returning `true`), plugin-registered commands/UI panels (the unified registries from ADR-0010 stay unused by plugins), and CI provisioning of a C compiler for `crates/plugin-host` was reasoned about (`platform-support.md` §2) but not verified against a real CI run in this session.
