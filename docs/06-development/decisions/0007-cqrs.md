# ADR 0007: CQRS and Read Projections Selection

## Context
The Terminal Workspace handles numerous real-time events. If the UI queries the active database or parses domain aggregates directly on every TUI redraw tick (60 FPS), the system will stutter, leading to high CPU usage. We need to decouple state updates from read actions.

---

## Decision
We select **CQRS (Command Query Responsibility Segregation)** with in-memory **Read Model Projections** for TUI rendering.

- **Command (Write Path)**: State mutations go through `Command -> CommandHandler -> Domain Event`.
- **Query (Read Path)**: The UI thread reads strictly from an optimized in-memory `DashboardReadModel`. When domain events are raised, a Projector updates this model instantly.

---

## Alternatives Considered

### Traditional CRUD (Active Record style)
- **Pros**: Simple code layout.
- **Cons**: Every panel draw requires querying SQLite tables or calling locks on complex aggregates. This blocks the UI thread, causing frames to drop during active Slack/GitHub API synchronization tasks. (Rejected).

---

## Consequences
- **Speed**: Renders are sub-millisecond, as data is read directly from memory fields.
- **Auditability**: Having all mutations mapped as explicit `Command` objects makes it easy to track actions and implement step-by-step local execution tracing.
- **Complexity**: Adds slight boilerplate (defining both commands, events, and projection logic). This overhead is justified by the requirement for smooth 60 FPS UI loops.

---

## Amendment (Phase 3: Write Path Only)

Phase 3 implements the **write path** of this ADR: `crates/commands::InMemoryCommandDispatcher` + `WorkspaceCommandHandler` route the existing `Command` enum to repository mutations and (where a matching frozen `Event` variant exists) publish resulting events.

The **read path** — `DashboardReadModel` and its Projector — is deliberately **not** built yet. It has no consumer: no phase has built the TUI that would read it. Building it now would mean guessing its shape without a real renderer to validate against. It belongs with whichever future phase adds the TUI Docking shell (`docs/02-architecture/ui.md`, ADR-0012), at which point the Projector can subscribe to `EventDispatcher` exactly like any other `EventHandler` from Phase 2.
