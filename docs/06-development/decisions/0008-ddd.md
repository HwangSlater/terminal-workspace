# ADR 0008: Bounded Context Boundaries (DDD)

## Context
As the workspace expands, integrating Slack, GitHub, Jira, and Google Calendar directly into TUI models will lead to a highly coupled "spaghetti" codebase. Modifying Slack code could break Calendar logic. We need clean domain boundaries.

---

## Decision
We enforce **Domain-Driven Design (DDD)** by separating the platform into 8 explicit **Bounded Contexts**:
1. `Workspace`: Coordinates UI focal nodes and active slots.
2. `Notification`: Core alerts and prioritization rules.
3. `Presence`: Developer online status caching.
4. `Task`: Local/Jira ticketing aggregates.
5. `Scheduler`: Calendar meetings, local agendas, Pomodoros.
6. `Integration`: External network connections and entity converters.
7. `Plugin`: WASM sandbox loading and executions.
8. `Assistant`: AI prompts, local tool contexts.

These boundaries are strictly isolated using separate sub-modules or crates.

---

## Alternatives Considered

### Single Monolithic Domain Model
- **Pros**: Quick implementation.
- **Cons**: High coupling. Models like `Notification` would have Slack-specific or GitHub-specific fields leaking into them, preventing the addition of new integrations (like GitLab or Discord) without rewriting core UI entities.

---

## Consequences
- **Low Coupling**: Changing integration protocols (e.g. updating Slack APIs) only impacts the `Integration` context, leaving the `Notification` domain untouched.
- **Strict Data Translation**: Integration adapters must map external data structures into pure Domain objects before broadcasting to the Event Bus.
