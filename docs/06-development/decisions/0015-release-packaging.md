# ADR 0015: Release Packaging via `cargo-dist`

## Context

`docs/01-product/product-requirements.md` §2.1 item 4 committed to prebuilt release binaries as the mechanism that makes "Zero Setup" true for end users, not just for the compiled artifact. `step4.md` scoped this out as Phase 4 and needed a concrete choice: hand-roll a GitHub Actions release job, or adopt an existing tool built for exactly this.

## Decision

Use **[`cargo-dist`](https://github.com/axodotdev/cargo-dist)**. Config lives in `dist-workspace.toml` (workspace-level) and `[package.metadata.wix]` in `crates/app/Cargo.toml` (Windows MSI metadata); the GitHub Actions workflow (`.github/workflows/release.yml`) is generated by `dist generate`, not hand-written — regenerate it via that command after config changes rather than hand-editing the YAML.

**Targets** (matching `docs/06-development/platform-support.md`'s Tier 1 list exactly, deliberately trimmed from `dist init`'s recommended default which also suggested `aarch64-unknown-linux-gnu` — not adding platform scope beyond what's already been decided and documented):
- `x86_64-pc-windows-msvc`
- `x86_64-apple-darwin`, `aarch64-apple-darwin`
- `x86_64-unknown-linux-gnu`

**Installers**: `shell` (Linux/macOS `curl | sh`), `powershell` (Windows), `msi` (Windows, via WiX — `crates/app/wix/main.wxs`, also generated).

## Alternatives Considered

### Hand-rolled `release.yml`
Full control, but every OS's build/package/checksum/upload step is ours to write and maintain. `cargo-dist` already produces all of that — tarballs, an MSI, shell and PowerShell installer scripts, and checksums — from one config file. Rejected: no concrete reason to carry that maintenance cost instead of using the tool built for this.

## Consequences

- `crates/app/Cargo.toml` gained `description` and `repository` (workspace-inherited) fields — `cargo-dist` requires both (the description for WiX MSI metadata, the repository for installer script URLs). These were missing before and are good `Cargo.toml` hygiene regardless of `cargo-dist`.
- **A git repository now exists** (it didn't before this ADR — `cargo-dist`'s tag-triggered release model is meaningless without one). No remote push has happened; `git init` was local-only.
- **Repository URL is a placeholder pending the user's actual GitHub repo**: `https://github.com/HwangSlater/terminal-workspace`. Update this and re-run `dist generate` if it ever changes.
- Per `step4.md`'s sequencing decision, this pipeline is validated via pre-release tags only (e.g. `v0.0.1-test`) against the current `crates/app` skeleton — **not** a public v0.1.0 announcement. The real public release waits for Phase 5 (TUI).
- Code signing (Windows/macOS) remains an open question — flagged in `step4.md`, deferred, not blocking this pre-release validation phase.
